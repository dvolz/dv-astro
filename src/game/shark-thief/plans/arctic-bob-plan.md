## Step 4 — Ice Patch Generation: `seedIcePatches()` in `spawn.ts`

**File:** `/Users/dvolz/Sites/dv-astro/src/game/shark-thief/spawn.ts`

Add this function after `spawnFrozenFish`. It seeds the board with ice patches during
Arctic level init, before anything else is placed.

```ts
export function seedIcePatches(count: number): void {
  // Shape definitions: [dx, dy] offsets from anchor cell
  const SHAPES: Array<Array<[number, number]>> = [
    [[0,0],[1,0]],                         // 1×2 horizontal
    [[0,0],[0,1]],                         // 1×2 vertical
    [[0,0],[1,0],[2,0]],                   // 1×3 horizontal
    [[0,0],[0,1],[0,2]],                   // 1×3 vertical
    [[0,0],[1,0],[2,0],[3,0]],             // 1×4 horizontal
    [[0,0],[0,1],[0,2],[0,3]],             // 1×4 vertical
    [[0,0],[1,0],[0,1],[1,1]],             // 2×2 square
  ];

  let placed = 0, attempts = 0;
  while (placed < count && attempts < 2000) {
    attempts++;
    const shape = SHAPES[Math.floor(Math.random() * SHAPES.length)];
    // Pick anchor so the full shape fits inside the grid
    const maxDx = Math.max(...shape.map(([dx]) => dx));
    const maxDy = Math.max(...shape.map(([, dy]) => dy));
    const ax = Math.floor(Math.random() * (GRID - maxDx));
    const ay = Math.floor(Math.random() * (GRID - maxDy));

    // Reject if any cell in the shape hits the shark spawn
    // (shark is already placed at gs.shark.x/y when this runs)
    const hits = shape.some(([dx, dy]) => {
      const cx = ax + dx, cy = ay + dy;
      return cx === gs.shark.x && cy === gs.shark.y;
    });
    if (hits) continue;

    for (const [dx, dy] of shape) {
      gs.iceCells[ay + dy][ax + dx] = true;
    }
    placed++;
  }
}
```

**Constraint check:** The design says the player spawn must not be ice. The shark is
placed before `seedIcePatches` is called (see `init()` in engine.ts:182–183), so the
shark position is known. The `hits` check above handles this.

The design also says ice is always passable (never a barrier), so there is no
connectivity/flood-fill requirement. Ice is a floor modifier only.

Do NOT call `seedIcePatches` from `init()`. Only call it from the Arctic-specific
init path in `checkDepthTransition` and `initAtDepth`. See step 7.

## Step 5 — Ice Slide Resolver: Helper Function in `engine.ts`

**File:** `/Users/dvolz/Sites/dv-astro/src/game/shark-thief/engine.ts`

Add a pure helper function, placed before `moveShark` (before line 213). This is
the slide resolver that both the shark path and the enemy AI will call.

```ts
/**
 * Resolves an ice slide starting from (startX, startY) in direction (dx, dy).
 * Returns the terminal cell — last valid cell before hitting a wall, coral,
 * or a non-ice cell.
 *
 * The starting cell itself is NOT re-checked — the caller already validated it.
 * Each intermediate cell is yielded to the `onCell` callback so the caller can
 * check for pickups, enemy contact, etc.
 *
 * Returns { x, y } of the terminal position.
 */
function resolveSlide(
  startX: number, startY: number,
  dx: number, dy: number,
  onCell?: (x: number, y: number) => "stop" | "continue",
): { x: number; y: number } {
  let x = startX, y = startY;
  while (true) {
    const nx = x + dx, ny = y + dy;
    // Hit board edge
    if (nx < 0 || nx >= GRID || ny < 0 || ny >= GRID) break;
    // Hit coral (Depth 2 barrier — not present in Arctic, but guard anyway)
    if (gs.coral[ny]?.[nx]) break;
    // Advance
    x = nx; y = ny;
    // Per-cell callback (check enemy contact, pickups if mid-slide collection is on)
    if (onCell) {
      const result = onCell(x, y);
      if (result === "stop") break;
    }
    // Stop if we've left ice — next cell is not ice, we're at the terminal
    if (!gs.iceCells[y][x]) break;
  }
  return { x, y };
}
```

**Gotcha:** The slide loop stops when the current cell is not ice (meaning the entity
slid off ice onto normal floor). It does NOT look one cell ahead for ice — it stops
at the first non-ice cell it lands on. This matches the design: "until the entity
would move into a wall, an impassable tile, or a non-ice cell — the entity stops at
the last valid cell before the blocking tile." Read this carefully: non-ice = stop
at that cell, wall/coral = stop at cell before.

The design doc says the start cell (where the shark lands on the fish) converts to a
1×1 ice tile before the slide begins. So `gs.iceCells[startY][startX]` will be true
when `resolveSlide` is called from the fish-collection path.

```ts
export function drawFrozenFish(
  ctx: CanvasRenderingContext2D,
  px: number, py: number, ps: number,
): void {
  ctx.save();
  const cx = px + ps / 2, cy = py + ps / 2;

  // Body — icy blue oval
  ctx.fillStyle = "#5ab4d8";
  ctx.beginPath();
  ctx.ellipse(cx, cy, ps * 0.38, ps * 0.22, 0, 0, Math.PI * 2);
  ctx.fill();

  // Tail — triangle pointing left
  ctx.fillStyle = "#4aa8cc";
  ctx.beginPath();
  ctx.moveTo(px + ps * 0.12, cy - ps * 0.18);
  ctx.lineTo(px + ps * 0.12, cy + ps * 0.18);
  ctx.lineTo(px + ps * 0.04, cy);
  ctx.closePath();
  ctx.fill();

  // Frosted overlay — white center highlight
  ctx.fillStyle = "rgba(255,255,255,0.45)";
  ctx.beginPath();
  ctx.ellipse(cx - ps * 0.06, cy - ps * 0.06, ps * 0.16, ps * 0.09, -0.4, 0, Math.PI * 2);
  ctx.fill();

  // Sparkle dot (top-right)
  ctx.fillStyle = "#ffffff";
  ctx.beginPath();
  ctx.arc(cx + ps * 0.22, cy - ps * 0.16, Math.max(1, ps * 0.05), 0, Math.PI * 2);
  ctx.fill();

  // Eye
  ctx.fillStyle = "#08181f";
  ctx.beginPath();
  ctx.arc(cx + ps * 0.24, cy - ps * 0.04, Math.max(1, ps * 0.04), 0, Math.PI * 2);
  ctx.fill();

  ctx.restore();
}
```

## Architectural Risks and Ordering Gotchas

**Circular import risk (high, will break at runtime):**
`ai.ts` imports nothing from `engine.ts`. `engine.ts` imports `moveEnemiesAI` from
`ai.ts`. If you put `resolveSlide` in `engine.ts` and try to import it into `ai.ts`,
you will create a circular module dependency. ES modules handle some cycles, but this
one will likely produce `resolveSlide is not a function` at runtime because of
initialization order. Use the `slide.ts` extraction described in step 8.

**Slide after fish collection before enemy AI runs (medium risk):**
The enemy spawned in step 6 (fish collection → spawn enemy) is pushed to `gs.enemies`
before `resolveSlide` runs. The freshly spawned enemy position comes from
`spawnEnemy()`, which has a `MIN_ENEMY_DIST` guard — the new enemy will be at least
5 cells from the shark's pre-collection position, but could be closer to the post-
slide terminal. This is correct behavior per design ("you triggered it, you die"),
but it means `spawnEnemy`'s distance guard relative to the original shark position
may feel unfair. Flag this to Zak.

**Enemy AI runs AFTER fish collection (no risk, just clarify):**
In `moveShark`, enemy AI (`moveEnemiesAI()`) runs at line 314:
```ts
if (gs.moveCount % 2 === 0) moveEnemiesAI();
```
This happens AFTER all pickup collection including the frozen fish block you're
adding. So the newly spawned enemy will not move until the next eligible move cycle.
The design says steps 3–4 happen before the next enemy turn — this is satisfied by
the existing structure.

**`checkDepthTransition` is called inside fish collection (medium risk):**
This matches exactly what coins, ammonites, and eggs do. It is safe. But be aware:
if the fish award pushes the score to exactly `depthEntryScore + 100`, the depth
transition fires immediately during the fish collection block, before `spawnFrozenFish`
runs. The transition handler resets `gs.frozenFish = null` (in step 7b), so
`spawnFrozenFish` at the end of the fish block would then place a fish in the new
depth. Since Arctic (depth 4) transitions into the Leviathan depth (5) and the cap is
`>= 5`, this is not a problem in practice.

**`gs.iceCells` must be initialized in `init()` before any spawn function runs (high):**
`spawnEnemy()` (after your step 8 edit) will read `gs.iceCells[ey]?.[ex]`. If
`gs.iceCells` is undefined, this will throw. The `?.` operator gives you null-safety
against `undefined` rows, but not against the array itself being undefined. Initialize
it in `init()` as described in step 7c. Do this BEFORE any call to `spawnEnemy()` in
`init()`. Looking at `init()` line 207: `gs.enemies = [spawnEnemy()]` — your
`gs.iceCells = ...` line at line 179 is before this, so the order is safe.

**`colorblindMode` path in `draw()` skips tile rendering (no risk, just awareness):**
When `gs.colorblindMode` is true (line 277), the tile and ice rendering is skipped.
Your ice patch renderer is inside the `else` branch and will correctly not run in
colorblind mode. The frozen fish renderer is outside the colorblind branch (it's
added after line 348) and WILL render in colorblind mode, which is correct — pickups
always render.

**`loadGame` duplication (pre-existing, but you must not make it worse):**
There are two `loadGame` functions — one in `persistence.ts` (line 82) and one in
`engine.ts` (line 396). Both must be updated with the new `iceCells` and `frozenFish`
restore logic (step 9). If you update only one, saves will work in one path but
silently fail in another. The `engine.ts` version is the one called by `navigation.ts`
on boot. Check which path calls which and update both.

**`saveGame` storage format (low risk, handled by fallback):**
The save key is `"sharkIosSave_v3"`. The fallback pattern `|| []` in the load
functions means existing saves will restore cleanly with empty ice arrays. Do not
bump the key version unless you want to invalidate all existing saves.
